## Context

- ADR-001 line 34 requires: every endpoint call recorded with agent identity, endpoint, params, status, timestamp; audit log "queryable for debugging and trust-building with clients." Line 77 summarizes Phase 6 as "audit log table, middleware, query capabilities."
- **THE GOVERNING BUSINESS NEED (Scott, 2026-05-29):** Ledgerix Pro handles client books in a litigious environment. It must be able to DEFINITIVELY defend itself when a past client claims "Ledgerix Pro did X/Y/Z to my books" ‚Äî by pulling exactly what was or was not done, per tenant, across the previous 7 years. This reframed the whole phase from "comprehensive logging" to **litigation-grade audit trail.**

**Critical terminology pin.** This arc is referred to as **"ADR-001 Pattern B Full ‚Äî Phase 6 (audit log, middleware, query capabilities)."** Its long-term-archival sub-piece is what the **Strategic Plan separately numbers as its own "Phase 6b (long-term data retention infrastructure ‚Äî 7-year archival, encryption-at-rest, legal hold)."** **THESE ARE THE SAME BODY OF WORK.** To avoid the collision, this WIP doc and all commits use the labels **6a / 6b / 6c** as defined below; whenever **"Phase 6b" appears in the Strategic Plan it maps to our 6c**. Never use a bare "Phase 6b" in commits for this arc ‚Äî use **"6c (archival)"**.

**Sub-phase labels (this WIP doc's vocabulary, used in all commits):**
- **6a** ‚Äî the queryable-audit half of ADR-001 Phase 6 (middleware completeness backstop + per-tenant query + fidelity/standardize logging). Split internally into **6a-0** (archive-before-delete hook ‚Äî fix the audit-destruction bleeding) and **6a-rest** (everything else under 6a).
- **6b** ‚Äî integrity / tamper-evidence on the (now archive-backed) audit trail.
- **6c** ‚Äî the archival half (Strategic Plan calls this "Phase 6b"). 7-year durable archive, encryption-at-rest, legal-hold mechanism. Split internally into **6c-code** (archival writer + format + encryption + legal-hold + lifecycle, against the existing StorageService abstraction) and **6c-infra** (real-world bucket provisioning + Railway prod config ‚Äî Scott's vendor/cost decision).

**KEY FINDINGS from code reads (2026-05-29), which drove the decisions:**

1. **`activity_log` IS the audit table** (no separate `audit_log`). Columns already capture ADR-001 line-34 fields: `actorType`, `actorId`, `agentId`, `action`, `entityType`, `entityId`, `details` (jsonb), `status` (success/failure), `createdAt`, `companyId` (nullable, 4c.5 Decision B). The WRITE surface `logActivity(db, input)` is live and called across ~15 route files incl. all accounting writes. So "table" + "logging" requirements are effectively MET.
2. **The genuine gap per ADR-001 line 34 is the QUERYABLE read side** ‚Äî only a narrow user-profile-activity query exists; no general per-tenant audit query.
3. **INTEGRITY THREAT: `activity_log` is NOT append-only** ‚Äî two code sites actively DELETE audit rows: `companies.ts` `remove()` (cascade across ~24 child tables to satisfy FK before deleting the company) and `agents.ts` `remove()` (same pattern). This directly contradicts 7-year retention. Confirmed NOT a GDPR/erasure mechanism (docs: all clients US-based, no EU/GDPR exposure; deletes are plain referential-integrity cascades).
4. **`activity_log.companyId`/`agentId` are FOREIGN KEYS** to `companies`/`agents` ‚Äî the audit is structurally chained to the live entity, which is WHY the cascade deletes it.
5. **A production-grade storage abstraction ALREADY EXISTS:** `server/src/storage/` with provider-registry + `local-disk-provider` + `s3-provider` (`@aws-sdk/client-s3`, configurable bucket / region / endpoint / prefix). Deploy docs confirm S3-compatible object storage (AWS S3, MinIO, R2) is the prod pattern. So durable external archival storage is a BUILT primitive, not greenfield.
6. **No archival bucket is provisioned in prod today** (Railway Postgres + Railway container backups, hourly/month-scale only ‚Äî nowhere near 7 years). Legal-hold mechanism named in Strategic Plan, not implemented.

If this didn't get done: a litigation event against Ledgerix Pro for past-tenant book changes could not be answered with definitive records ‚Äî exactly the trust-defining capability the Trust Tenet exists to support.

### 6c-code design decisions (locked 2026-05-29)

- **Decision P ‚Äî Archival format & granularity (P1):** Per-tenant, per-time-window JSONL objects. Key structure follows the existing StorageService convention (`{companyId}/audit/{year}/{month}/...`); each line is one activity_log row WITH DENORMALIZED IDENTITY (company name + agent name as they were at the time of the action) so the record stands alone and is legible after the live company/agent entity is deleted. Rejected: per-row objects (millions of tiny objects, operationally/cost prohibitive); single rolling per-tenant object (append requires rewriting the whole object ‚Äî fights immutability and any future Object Lock). Denormalized identity is non-negotiable for the litigation use case. Locked.
- **Decision Q ‚Äî Encryption approach (Q1, app-level):** Encrypt the JSONL buffer with AES-256-GCM before putFile, mirroring the proven secrets pattern (`server/src/secrets/local-encrypted-provider.ts`: random IV per object, scheme-tagged, keyed by a master key). Chosen over S3 server-side encryption (SSE) because app-level is STORAGE-BACKEND-INDEPENDENT ‚Äî the archive is encrypted identically on local-disk (dev/test) and S3 (prod), so the real encryption path is exercised in dev and doesn't depend on per-bucket SSE config. GCM's auth tag also gives tamper-DETECTION on the archive (a down-payment on 6b integrity).
  - **CRITICAL CAVEAT ‚Äî KEY SURVIVABILITY IS LOAD-BEARING (added to 6c-infra pending):** App-level encryption means the master key is required to decrypt archives for the full 7 years. A lost key = unrecoverable archives = no producible evidence in litigation ‚Äî the exact failure the whole arc exists to prevent. Therefore 7-year master-key management/escrow is a MANDATORY 6c-infra obligation, NOT optional. Q1 is correct ONLY if key survivability is guaranteed. (Q3 ‚Äî app-level + SSE belt-and-suspenders ‚Äî was considered and deferred; SSE does not mitigate app-level-key loss anyway, so the real mitigation is key escrow, tracked below.)
  Rejected: Q2 (SSE only ‚Äî backend-dependent, plaintext in app, weaker tamper story). Locked (with the key-management obligation).
- **Decision R ‚Äî Legal-hold mechanism (R1, app-level registry):** A DB table (e.g. `legal_holds`: tenant/companyId, reason, placedAt, placedBy, liftedAt) that the archival lifecycle/destruction logic consults ‚Äî an archive under an active hold is never destroyed regardless of 7-year age. Chosen over S3 Object Lock (R2) because the existing S3 provider does NOT expose Object Lock/Retention/Legal Hold (only put/get/head/delete), so R2 would require extending the provider AND depend on un-built bucket config; and because the destruction decision is app-level anyway, so the hold check belongs beside it as a simple, testable, backend-independent DB lookup. Legal hold is a business state ('this tenant is under subpoena/IRS audit/dispute'), naturally a DB record. Rejected: R2 (S3 Object Lock ‚Äî couples to un-built infra); R3 (defer hold entirely ‚Äî the destruction logic should be born hold-aware). Locked.

### Decision S ‚Äî Point-in-time identity capture (locked 2026-05-29)

**Decision: Option 2 ‚Äî capture company name + agent name into the activity_log row AT WRITE TIME**, so every audit record carries the identity as it was at the moment of the action (not as re-derived later).

Rationale: P1 requires denormalized identity 'as it was at the time of the action' so the archive stands alone. A read confirmed logActivity does NOT snapshot names today ‚Äî the row stores companyId/agentId (FKs) + an action-specific details jsonb; names live only in the mutable companies/agents tables. An archive-time join would collapse all history to the name as-of-archive/deletion, losing mid-engagement renames. For a litigation record where fidelity is the point, 'agent X did Z when the company was named Y-at-the-time' is stronger. Critically, there are NO client audit rows yet (no clients until the system is complete), so backfilling logActivity to snapshot identity going forward has ZERO legacy-data gap ‚Äî every real client row gets point-in-time identity from day one. This is a rare clean window to do it right at the source.

Rejected: Option 1 (archive-time join ‚Äî simpler, but loses mid-engagement identity history); Option 3 (hybrid/defer ‚Äî no reason to defer given the clean no-data window).

**Storage shape (implementation form of S):** point-in-time identity is stored as DEDICATED, nullable columns on activity_log (e.g. `company_name_snapshot text`, `agent_name_snapshot text`), NOT inside the details jsonb. Reasoning: (a) details is run through sanitizeRecord + redactCurrentUserValue at write time ‚Äî a company/agent name could be wrongly redacted by the username-censor; identity metadata must not be subject to that. (b) Point-in-time identity is structural audit metadata, not action detail ‚Äî keeping it as typed columns is queryable and explicit, and the archive-writer reads it directly rather than digging through a blob. Cost: one additive nullable migration. Locked.

**Work-order consequence:** logActivity identity-capture is now a FOUNDATIONAL 6c-code piece, sequenced BEFORE the archive-writer (the writer reads what logActivity stores). logActivity must look up + store the current company/agent name at insert time; callers that pass companyId/agentId get the snapshot captured automatically inside logActivity (callers unchanged where possible).

**SCOPE NARROWED 2026-05-29 (consequence of T REVISED).** S's intent ('every real client audit row carries point-in-time identity') is narrowed to: every ACCOUNTING-WRITE audit row carries point-in-time identity (the litigation-defense-of-books surface). Non-accounting audit rows leave the snapshot columns NULL. The dedicated nullable columns added in 2a (commit 6a90fea6) remain correct and unchanged ‚Äî nullable was already the right call; this only narrows WHICH rows populate them.

### Decision T ‚Äî logActivity identity-capture mechanism (locked 2026-05-29)

**Decision: Option 3 (hybrid) ‚Äî optional caller-supplied point-in-time names, with a lookup fallback when not supplied.**

Mechanism: extend LogActivityInput with optional `companyNameSnapshot?` / `agentNameSnapshot?`. In logActivity: if a name is supplied by the caller, store it directly (no query). If NOT supplied AND the corresponding id is present (companyId / agentId non-null), look it up (query companies/agents by id) and store the result. System-scoped rows (companyId null) and rows without agentId store null snapshots, as today.

Rationale ‚Äî chosen over Option 1 (always lookup) and Option 2 (caller-supplied only):
- A conversation-search confirmed a real PATTERN of deferred-performance / v1-simplification decisions already accumulated on hot/agent paths: the unhinted getTransactionById multi-type probe loop (hintedType fast-path dropped), rate-limiting deferred to v2, caching deferred to v2, payments remainingBalance/invoiceStatus simplified out. Scott explicitly flagged concern about PILING ON further deferred-perf debt.
- logActivity is a HOT path: 142 call sites. Option 1 (always lookup) would add up to 2 extra DB round-trips PER audit write across all 142 sites ‚Äî adding two more items to the known deferred-perf stack, on a constantly-firing path.
- Option 3 avoids that: the litigation-critical callers (accounting write paths) already hold the company/agent object, so they supply names with ZERO added queries ‚Äî accurate point-in-time identity on exactly the records that matter most. The lookup fallback guarantees COMPLETENESS (no silent null-snapshot gaps) for the long tail of callers that don't supply names.
- Trade accepted: Option 3 is more code (two paths in logActivity) than Option 1. This deliberately trades invisible-future-perf-debt for visible-testable-complexity-now ‚Äî the right trade given the established pattern of deferred costs.

Rejected: Option 1 (always lookup ‚Äî simplest, but adds 2 queries √ó 142 hot-path sites to the deferred-perf pile); Option 2 (caller-supplied only ‚Äî cheapest, but risks silent null-snapshot gaps in litigation-critical records if a caller isn't updated). Locked.

**Implementation note:** As callers are NOT all updated at once, the accounting write paths (litigation-critical) should be updated to supply names in the SAME work as the logActivity change or immediately after; the fallback covers correctness in the interim. No client rows exist yet, so there is no legacy gap regardless.

**REVISED 2026-05-29 (premise corrected, scope narrowed).** Verify-step finding: the litigation-critical accounting write callers (accounting.ts) hold companyId + req.actor.agentId ‚Äî IDs ONLY, no name and no loaded company/agent object at the logActivity call site. T's original Option-3-hybrid premise ('accounting callers supply names with zero added query') is therefore FALSE ‚Äî they cannot cheaply supply names. Revised decision: do NOT add the optional caller-supplied-name machinery to the general logActivity path, and do NOT impose a lookup on the 142-site hot surface. INSTEAD: confine point-in-time identity capture to the low-volume accounting write paths only ‚Äî those callers perform the company/agent name lookup explicitly (low volume ‚Üí the 142-site perf concern that drove T away from always-lookup does NOT apply here) and pass the resolved names to logActivity via the optional snapshot fields. Non-accounting audit rows store NULL snapshots (they don't carry the litigation-defense-of-books requirement). Net effect: the litigation fidelity lands exactly where the requirement bites (the books), the 142-site hot path gets ZERO added queries, and the implementation is simpler than the abandoned general hybrid. The optional companyNameSnapshot/agentNameSnapshot fields on LogActivityInput are RETAINED (the accounting callers use them to pass resolved names); what's dropped is the general in-logActivity lookup-fallback for the whole surface.

### 6a-AUDIT scope decisions (locked 2026-05-29)

Validated finding: accounting book-changes are logged to activity_log NOWHERE. Route-side (success 200 / approval-request 202 / failure 4xx-5xx) ‚Äî none logged. Replay-side (approve() ‚Üí executeApprovedAccountingWrite returns write_executed / write_failed_replay) ‚Äî captured into `const result` but never logActivity'd ('for Phase 4c.4 we just log' = pino stdout only). The dispatcher's audit-data return shape is a designed-for-but-unimplemented seam. 6a-AUDIT instruments BOTH sites.

- **Decision U ‚Äî Which events log (U3, everything):** Every book-touching event gets an activity_log row: route success (status=success), route approval-request (the 202 queued events), route failure (status=failure, write attempted-but-rejected), replay write_executed (status=success), replay write_failed_replay (status=failure). Rationale: the litigation requirement is 'pull exactly what was OR WAS NOT done' ‚Äî a queued request and a rejected attempt are as legally valuable as a success ('was not done, here's why' shows diligence + the safety system working). activity_log.status already supports success/failure. These are low-volume write routes, NOT the 142-site hot path, so the extra log calls carry no hot-path perf concern. Rejected: U1 (book-changes only), U2 (book-changes + approval-requests, omits failures).
- **Decision V ‚Äî Route-side log placement (V2, after the result):** Log off the final result.status returned from withIdempotency, once per outcome, rather than inside the idempotency callback (avoids double-logging vs. cached-replay semantics). CAVEAT (implementation-time): failure branches (throw badRequest / throw err) escape the callback AND withIdempotency entirely ‚Äî they cannot be logged 'after the result' and need a try/catch around the operation or logging at the throw sites. The exact mechanism depends on withIdempotency's replay behavior and must be confirmed by reading withIdempotency BEFORE writing the route-side calls (so retries don't double-log and failures aren't missed). Rejected: V1 (log inside the idempotency callback ‚Äî double-log risk on replay).
- **Decision W ‚Äî Replay-side log placement (W1, in approve() after dispatcher):** Log in approvalService.approve() after executeApprovedAccountingWrite returns, reading result.action (write_executed ‚Üí success log; write_failed_replay ‚Üí failure log). This completes the exact seam the existing code comment describes ('available here ... a separate activity_log entry'). approve() has the approval context (companyId, decidedByUserId, payload). Keeps the dispatcher a pure execute-and-report function. Rejected: W2 (log inside the dispatcher ‚Äî mixes execution with audit-logging concern).
- **Decision X ‚Äî Point-in-time identity threading (X1, resolve once per scope):** Resolve company name + agent name ONCE per request (route-side, from companyId + req.actor.agentId) / once per approval (replay-side, from approval.companyId + payload requestedByAgentId), reusing the existing companyService/agentService helpers, passing via the optional snapshot fields shipped in 2b-i (commit c64bf177). Low-volume ‚Üí single resolve each. Rejected: X2 (resolve per log call).

**V2 caveat RESOLVED 2026-05-29 (withIdempotency read).** Confirmed: withIdempotency returns cached response WITHOUT re-running work() on replay (book-change occurs once, on original execution); a work() throw propagates out uncached to the global error handler. Therefore route-side logging:
- Success (200) + approval-request (202): logged AFTER withIdempotency returns, ONLY when result.replayed === false, keyed on result.body.status ('success' ‚Üí success log; 'pending_approval' ‚Üí queued-request log). Replays log NOTHING (books didn't change again; the original execution already logged).
- Failures: wrap the withIdempotency call in try/catch; on catch, write a status:'failure' activity_log row capturing the error code/reason, then RETHROW so the global error handler still produces the HTTP response.

- **Decision Y ‚Äî Which failures log (Y2, post-validation only):** Only failures where a write was genuinely ATTEMPTED log (upstream QBO/Xero errors, type-not-categorizable, pricing-not-configured ‚Äî these occur inside/after withIdempotency). Pure INPUT-VALIDATION rejections (malformed body, bad UUID, missing field ‚Äî thrown at the top-of-handler validation block BEFORE withIdempotency runs) are NOT logged: nothing touched the books, and logging them floods the audit trail with garbage-input noise. The litigation record is about book-events, not request-events. The try/catch therefore wraps from the withIdempotency call onward, NOT the input-validation block. Rejected: Y1 (log all failures incl. input-validation ‚Äî buries book-signal in input noise).
- **Decision Z ‚Äî Identity resolution placement (Z1, once up-front):** Resolve company name + agent name ONCE per request after assertCompanyAccess (companyId is validated + access-checked by then), before the withIdempotency call, so the resolved snapshots are available to BOTH the success/approval after-path AND the failure catch-path. Reuse across whichever log fires. One company + one agent lookup per request (low-volume routes). Rejected: Z2 (lazy per-log resolution ‚Äî saves nothing, complicates the catch path).

### 6c archive-writer decisions (locked 2026-05-29)

Building blocks confirmed by read: secrets module exposes AES-256-GCM `encryptValue(masterKey, value:string)‚Üí{scheme,iv,tag,ciphertext}` (base64) + `decryptValue` ‚Äî proven, authenticated, takes a STRING (JSONL is text, composes directly). `StorageService.putFile` requires a non-empty Buffer body + builds `{companyId}/{namespace}/{date}/...` keys; `getObject(companyId, objectKey)` is company-scoped via `ensureCompanyPrefix` (tenant-isolated retrieval by construction). activity_log query mirrors `db.select().from(activityLog).where(and(eq(companyId), gte(createdAt, from)))` with the snapshot columns (companyNameSnapshot/agentNameSnapshot) available.

- **Decision AA ‚Äî Dedicated archive encryption key (AA2), NOT the secrets master key.** The archive uses its OWN key (e.g. `PAPERCLIP_ARCHIVE_MASTER_KEY`), separate from `PAPERCLIP_SECRETS_MASTER_KEY`. Rationale (ties directly to the Q1 7-year key-escrow obligation): the secrets key protects LIVE operational secrets (OAuth tokens) and SHOULD be rotatable (rotate ‚Üí re-encrypt live secrets ‚Üí done). The archive key must remain valid for 7 YEARS to decrypt old archives for litigation ‚Äî it effectively cannot rotate without re-encrypting the entire archive history. COUPLING them means either you can never rotate the secrets key without touching 7 years of archives, OR rotating the secrets key silently destroys the ability to decrypt old archives ‚Äî the catastrophic "unrecoverable evidence" failure flagged under Q1. Separating the keys makes the 7-year escrow a clean, independent no-rotate discipline. Reuse the SAME crypto implementation (mirror `encryptValue`/`decryptValue`'s AES-256-GCM pattern) but with the dedicated key. Rejected: AA1 (reuse secrets master key ‚Äî couples key lifecycles; latent litigation-defense landmine).
  - **6c-infra consequence:** 6c-infra must provision `PAPERCLIP_ARCHIVE_MASTER_KEY` (32-byte) with its OWN 7-year escrow + explicit NO-ROTATE discipline, distinct from the secrets key. This SUPERSEDES the generic "master-key escrow" pending item under Q1 ‚Äî it's specifically the archive key.
- **Decision BB ‚Äî Writer + retrieval together (BB2).** This first 6c commit ships BOTH the archive-writer (query tenant activity_log rows for a window ‚Üí serialize JSONL ‚Üí GCM-encrypt with the archive key ‚Üí `StorageService.putFile`) AND the retrieval/decrypt path (`getObject` ‚Üí decrypt ‚Üí parse JSONL back to rows). Rationale: the cleanest verification is a ROUND-TRIP (write ‚Üí read back ‚Üí decrypt ‚Üí assert rows match); the writer can't be meaningfully tested without the reader, and they share the format/crypto contract. The legal_holds-aware lifecycle/destruction logic is SEPARATE and later ‚Äî this commit is put + get only. Rejected: BB1 (writer-only first ‚Äî can't round-trip-verify).
- **Decision CC ‚Äî Full-row JSONL (CC1).** Each JSONL line = one activity_log row with ALL columns (id, companyId, actorType, actorId, action, entityType, entityId, agentId, runId, details, status, companyNameSnapshot, agentNameSnapshot, createdAt). Litigation archive ‚Üí capture everything; a missing field years later is unrecoverable; storage is cheap. Rejected: CC2 (trimmed subset).

### 6a-0 delete-hook decisions (locked 2026-05-29)

- **Decision DD ‚Äî Archive scope: company-deletion ONLY (DD3).** The 6a-0 hook archives a tenant's activity_log on `companies.remove()` ONLY. `agents.remove()` does NOT trigger an archive. Rationale: the litigation requirement is about the CLIENT'S BOOKS (company-scoped, 7-year tenant record); company-deletion is the tenant-record-preservation event. Agent-deletion is a DIFFERENT concern and is ALREADY MITIGATED by the point-in-time identity work (S/T): every accounting-write activity_log row already carries `agentNameSnapshot`, so even after an agent row is deleted, the company's audit trail still attributes the action to the named agent. The book-touching events survive the agent's deletion via the company trail. Archiving the whole company trail on every agent-delete (DD1) over-captures + duplicates; building agent-scoped archival (DD2) is real work for a preservation question not established as litigation-relevant.
  - **ACCEPTED GAP (explicit, not an oversight):** `agents.remove()` permanently destroys that agent's activity_log rows (agentId-matched + run-linked) UN-ARCHIVED. Accepted because (a) the litigation-relevant book-events are also reflected in the company's trail with snapshot identity, and (b) company-deletion archives everything when the tenant actually leaves. If an agent's FULL operational history is ever deemed part of the 7-year record, the clean later addition is to extend the writer to accept an agent-scoped predicate. NOT doing that now.
  Rejected: DD1 (archive whole company on agent-delete ‚Äî over-captures/duplicates), DD2 (agent-scoped archival ‚Äî unbuilt writer capability for a non-established need).
- **Decision EE ‚Äî Archive-failure blocks deletion; archive runs BEFORE the delete transaction (EE3).** The 6a-0 hook archives the company's activity_log BEFORE entering the existing `companies.remove()` delete transaction. If the archive SUCCEEDS, the delete transaction proceeds. If the archive FAILS (storage down, key missing, etc.), the deletion is ABORTED ‚Äî the caller gets an error, the tenant + its rows remain intact, and the operation is retry-able. Rationale (Trust Tenet): losing the audit trail is WORSE than failing a deletion. Archiving the books-record is a HARD PRECONDITION of deleting a tenant ‚Äî you cannot destroy a tenant's 7-year book-history until it is durably archived. Archive-before-txn (vs inside-txn) avoids holding a DB transaction open across a storage round-trip and avoids orphan-archive-on-rollback.
  - **CRITICAL INVARIANT:** archive-failure must NEVER proceed-and-delete (that would silently destroy the exact record the arc preserves). The hook does NOT catch-and-continue on archive failure; it lets the failure propagate to abort the deletion.
  Rejected: EE1 (archive inside the txn ‚Äî holds txn open across storage I/O; orphan-on-rollback; though atomically cleaner), EE2 (archive inside txn but failure does NOT block delete ‚Äî DANGEROUS: silently loses the record on archive failure).

**6a-rest scope narrowed (2026-05-29):** 6a-rest is scoped to the QUERY/retrieval capability only. The originally-bundled completeness-backstop middleware and logging-fidelity standardization are split into separate later pieces ‚Äî they have different characters (preventive infra for not-yet-existing routes; cleanup) from the retrieval capability, and bundling risked an unfocused piece. The query is the direct, high-value continuation: it makes the now-durable trail actually usable for litigation/trust ("pull exactly what was done to a tenant's books").

### 6a-rest-QUERY decisions (locked 2026-05-29)

Read findings: archiving happens at exactly one site (the delete-hook), so live `activity_log` rows and archived objects are DISJOINT (a tenant is live-only or deleted-only, never both). And a GAP: nothing persists a deleted tenant's archive objectKey, so the archive is durable-but-unfindable (no `StorageService.list` capability; `readArchive` needs a known key).

- **Decision FF ‚Äî audit_archives manifest table (FF1).** Add a persistent `audit_archives` table (companyId, objectKey, archivedAt, rowCount, sha256, plus window-from/window-to) that the delete-hook writes a row to WHEN it archives. This table is NOT cascade-deleted with the company ‚Äî it is the INDEX TO the deleted tenant's archived remains, and must outlive the tenant. Retrieval for a deleted tenant: query `audit_archives` by companyId ‚Üí get objectKey(s) ‚Üí `readArchive`. Rationale: without this, archives are durable but unfindable ‚Äî the litigation answer ("produce this former client's records") is undeliverable. The manifest is the honest completion of preservation: archived AND indexed. Rejected: FF2 (deterministic/listable object keys ‚Äî couples retrieval to key conventions, needs an unbuilt `StorageService.list`, S3 listing consistency quirks); FF3 (defer deleted-tenant retrieval ‚Äî leaves preservation with a quiet hole: durable-but-unfindable archives).

**Manifest implementation shape (II1/JJ1/KK1, locked 2026-05-29):**
- **II1 ‚Äî companyId is a PLAIN uuid column, NO foreign-key reference to companies.** The manifest's defining property is that it OUTLIVES the company it indexes (it's the pointer to a deleted tenant's archive), so it must NOT be relationally bound to the company row. A FK would either block the company delete (ON DELETE NO ACTION) or, if cascaded, delete the manifest itself ‚Äî defeating it. Plain indexed companyId records the association without a constraint that blocks or cascades. (Deliberately keeping a pointer to an entity that will cease to exist; FK integrity is not wanted here.)
- **JJ1 ‚Äî skip the manifest row for empty archives.** When `archiveActivityForCompany` returns `{objectKey:null, rowCount:0}` (no audit rows ‚Üí no object written), write NO manifest row. A manifest row indexes an archive OBJECT; with no object there is nothing to point at. The deleted-tenant query treats "no manifest row" as "no archived audit", which is accurate. (A deletion-with-zero-history tombstone is a different feature, not an archive index ‚Äî not built here.)
- **KK1 ‚Äî manifest write is under the EE3 blocks-deletion invariant.** The hook order becomes: archive (produces objectKey) ‚Üí write manifest row (records objectKey) ‚Üí delete transaction. Both the archive AND the manifest write run OUTSIDE the delete txn with NO try/catch; a failure in either propagates and ABORTS the deletion. Extends EE3's "never proceed-and-lose" to "never proceed without a findable index": you cannot delete a tenant until its audit is both archived AND indexed. EE3/DD3 contract unchanged ‚Äî this extends the locked interface.

**5b query implementation (LL1/MM1/NN1, locked 2026-05-29):**
- **LL1 ‚Äî either/or resolution; "neither" returns EMPTY, not 404.** `getById(companyId)`: if the company EXISTS ‚Üí run a dedicated full-row live query (NOT `activityService.list`, which is a heavyweight UI feed; a clean `select().from(activityLog).where(companyId).orderBy(createdAt asc)` mapped to `ArchivedActivityRow`). If the company is GONE but `audit_archives` rows exist ‚Üí archive path (MM1). If NEITHER (no company, no manifest) ‚Üí return an EMPTY result, NOT a 404: "produce everything on record for tenant X" has a legitimate answer of "nothing on record", which is information, not an error. Edge: if a company EXISTS and somehow also has manifest rows, the LIVE path wins (live rows are the complete current trail; windowed archiving of a live tenant is never called).
- **MM1 ‚Äî read ALL manifest rows, concatenate.** A deleted tenant may have >1 `audit_archives` row (future windowed archiving / retried deletion). Read every manifest row for the companyId, `readArchive` each, merge the rows ordered by `createdAt`, return the union. Honoring all index entries is the honest "produce everything" read; reading only the latest would silently drop archives. (Today one row per tenant, so this is near-free future-proofing against a silent-loss bug.)
- **NN1 ‚Äî route returns rows + a `source` provenance discriminator; board-gated; NO `assertCompanyAccess`.** Response shape: `{ source: 'live' | 'archived' | 'none', rows: ArchivedActivityRow[] }`. The `source` field is litigation provenance (live operational DB vs sealed encrypted archive).
  - **CRITICAL AUTHZ NOTE:** this route uses `assertBoard` ONLY ‚Äî it must NOT use `assertCompanyAccess` (the sibling `/companies/:companyId/activity` route does). `assertCompanyAccess` would FAIL for a DELETED company (no company row to check access against), breaking retrieval for exactly the deleted-tenant case this feature exists for. The board-level gate is the correct and intentional authorization; a future change adding `assertCompanyAccess` here would be a regression that disables deleted-tenant audit retrieval. Route is operator/litigation-facing, NOT agent- or client-facing.
